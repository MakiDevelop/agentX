## 1. 專案目標總覽

### 原始目標
讓 agentX 成為一個真正「合格」的 Agent CLI（對標 Claude Code、Codex、Gemini CLI、Grok Build），特別是 headless 長任務的穩定性與可維護性。

### MT22 核心目標
解決長期存在的「雙任務系統分裂」問題：
- 舊系統：`task.py` + `TaskState`（單一任務）
- 新系統：`tasks.py` + `.agentx/tasks.json`（多任務清單，AgentSession 主力）

最終目標：讓新多任務系統成為**唯一真相來源**（Single Source of Truth），舊系統徹底退場。

## 2. 當前整體進度評估（我的標準）

**對 v0.3.0「可用且有尊嚴」的距離：約 55-65%**

### 已大幅達成的

- **Dispatch Refactor (Wave 1)**：已大幅完成（37 個 handler 已遷移）。
- **MT22 過渡層強化**：
  - `cli.py` 已完全移除對 `TaskState` 的直接 import。
  - Legacy 相容層已收斂到單一受控 helper：`_get_legacy_task_if_exists()`（已搬到 `tasks.py`）。
  - 該 helper 經過多輪「逐一強化」（防禦性、資料品質、正規化、可追溯性、警告機制）。
  - 所有公開舊函式（`load_task`、`start_task` 等）已加入 `DeprecationWarning`。
  - 啟動時若偵測到舊資料，會給使用者明確提示。
  - 新增診斷工具：`has_legacy_single_task()`、`get_task_migration_status()`。
- **遷移策略**：已定為「策略 B」（遷移成功後將舊 `task.json` 備份為 `task.json.bak.*`）。
- **使用者過渡體驗**：已有基本提示與警告。

### 仍需推進的主要項目（優先序由高到低）

1. **遷移機制可靠性**（最高風險）
   - `migrate_single_task_if_needed()` 測試仍需再補強到「敢移除舊系統」的等級。
   - 目前測試對常見與重要邊界情境已有較好信心，但還不夠完整。

2. **剩餘三處受控 legacy fallback 呼叫點收斂**
   - `print_config`
   - `build_handoff`
   - 啟動流程（相對最乾淨）
   - 目前已完成「檢查模式統一」+ helper 抽取 + TODO 標註。
   - 下一階段是進一步簡化或為移除做準備。

3. **`test_task.py` 技術債處理**
   - 這是目前最大的單一技術債。
   - 幾乎完全在測試舊系統。
   - 已加上強力 deprecation header + TODO。

4. **過渡期文件完整性**
   - 需要更清楚的「Migration Guide」或「v0.3.0 舊系統使用注意事項」。
   - handoff 文件內的 M 清單需要持續更新。

5. **正式 deprecation + 移除時間表**
   - `task.py` 整個模組的 deprecated 宣告還可以再加強。
   - 需要給出明確的移除版本或條件。

## 4. 重要設計決策紀錄（MT22）

- **策略 B**：遷移成功後將舊 `task.json` 備份為 `task.json.bak.*`（而非直接刪除）。（使用者最終選擇）
- **Legacy 讀取統一**：所有讀取舊單一任務的行為，都必須經過 `_get_legacy_task_if_exists()`。
- **過渡期原則**：寧可多保留備份與相容，也不要讓使用者資料在過渡期間有風險。
- **可觀測性優先**：大量加入 warning、提示、診斷工具，讓過渡過程可被觀察。
